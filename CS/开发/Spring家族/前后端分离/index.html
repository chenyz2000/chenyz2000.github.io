前后端分离

Posted on 2020-07-30 | Post modified: 2020-08-04 | In Spring家族

浏览器作为客户端，向服务器请求获得一段字符串（即HTML），之后经过浏览器的规则识别进行渲染

Spring Boot+Vue

传统的Java Web开发中，前端使用JSP开发，后端使用Java语言开发

JSP不是后端独立完成的，是由前端开发HTML，再由后端转成JSP

前后端开发的内容互相影响，互不理解

前后端耦合高，开发效率低

但是随着浏览器端展现和交互的要求越来越高，以及浏览器近些年性能和标准长足的发展，服务器端由JSP产生简单的HTML和CSS/JS已经无法满足需要。

所以采取前后端分离，服务器端不再负责任何展现和交付的工作，这些工作完全由前端独立完成。服务器端只通过接口给前端提供数据服务

前端通过Ajax请求访问后端的数据接口，将model展示到View中

这要求前后端商量好接口文档（url、参数、数据类型），之后就可以独立开发

前端使用假数据测试

传统开发：单体式应用。客户端通过访问服务端应用，服务端中的JSP和Java Server（无论是Handler、sevlet也好，总之是Controller层）互调，JSP请求Controller里的数据，Controller将数据解析到JSP里，最后拼接起来之后返回给客户端

前后端分离：单体应用分为前端和后端两个应用。客户端访问前端应用端口，前端应用访问后端应用端口，两者不同端口。前端应用里是html、css、js等代码，前端通过Ajax调用基于restful结构的后端数据接口，后端应用里是Java Server，Java Server响应前端给前端json数据，前端再把数据解析到页面中

Vue开发

vue开发，命令行创建vue项目，在idea中下载vue插件之后可以用idea打开，并编译项目，可以运行因为其中内嵌了一个服务器，运行命令npm run server

vue应用只有一个主窗口，src/app.vue是主窗口，之后的操作都是在其中进行动态更新组件。
组件放在src/views/中（当然放在component也可以，只要之后能够成功引入），通过路由的映射将app.vue中的名称和资源关联，路由放在src/router/中，无论浏览器访问的是什么url，主窗口并不改变，改变的只是其中的组件

使用vue，改了代码也不需要重新启动项目，可以动态更新页面

注解@entity标注实体类，和数据库的表相绑定。其中的属性加上@Id表示主键。实体类的每个对象是表里的一个记录。

HTTP

HTTP，超文本传输协议，基于TCP/IP通信协议来传递数据，用于前端和后端的交互中

浏览器的Ajax api，或者iOS/Android的http client，Java的commons-httpclient/okhttp或者是curl，postman之类的工具发出来的请求还能用在后端各个子服务的调用中（即当一种RPC协议使用）

HTTP客户端（浏览器）通过URL向HTTP服务端（WEB服务器）发送请求，服务端根据接收到的请求向客户端发送响应信息

HTTP使用统一资源标识符URI来传输数据和建立连接

HTTP特点：

  • 每次连接只处理一个请求。服务器处理完客户的请求，并收到客户的应答后，即断开连接
  • 只要客户端和服务器知道如何处理的数据内容，可以发送任何类型的数据
  • 对于事务处理没有记忆能力，若后续处理需要先前的信息则必须重传

消息格式

请求消息由请求行（request line）、请求头部（header）、空行和请求数据组成

响应消息由状态行、消息报头、空行和响应正文组成

请求方法

有八九种请求方法，其中最常用的是GET和POST，多数网站只支持这两种

区别
  • GET后退按钮/刷新无害，POST数据会被重新提交（浏览器应该告知用户数据会被重新提交）。
  • 与 POST 相比，GET 的安全性较差，因为所发送的数据是 URL 的一部分。在发送密码或其他敏感信息时绝不要使用 GET ，POST 比 GET 更安全，因为参数不会被保存在浏览器历史或 web 服务器日志中。
  • GET的参数在 URL 中对所有人都是可见的、会保留在历史中。POST的数据不会显示在 URL 中、不会保留在历史中。
  • GET书签可收藏、能被缓存，POST为书签不可收藏、不能缓存。
  • GET对数据长度有限制，当发送数据时，GET 方法向 URL 添加数据；URL 的长度是受限制的（URL 的最大长度是 2048 个字符）。POST无限制。
  • GET只允许 ASCII 字符。POST没有限制，也允许二进制数据。
适用情景

get用来获取html、css、图片等资源，post用来提交form表单，并得到一个结果的网页

查找时使用get，增添、修改（相当于删除再添加）服务器的文件或数据时使用post，删除时有说用get的有说用post的

向服务器发送大量数据时使用post

get更简单也更快，并且在大部分情况下可以使用

编码类型

  • text/xxx文本相关的格式，有html、plain、xml
  • application/xxx媒体相关格式，有html+xml即XHTML格式、xml、atom+xml、json、pdf、msword即Word文档格式、octet-stream即二进制流数据（如常见的文件下载）、x-www-form-urlencoded即form表单默认编码
  • key-value格式
  • multipart/form-data即二进制流，文件上传
  • image/xxx图片相关格式，有gif、jpeg、png

前后端数据交互

前后端数据交互：表单，Ajax

提交请求

表单用post/get方式提交

get是将键值对拼接到url的后面，即请求数据直接放在url中，通过访问或跳转，向服务器发送url请求（地址栏、超链接访问都属于get提交方式 ）

post是参数直接发送给后台，数据放在请求体body（或称HttpEntity，主体entity body）中，不是放在url中

根据Http的请求方法对应的数据传输能力，把Http请求分为Url类请求和Body类请求，Url类请求包括但不限于GET、HEAD、OPTIONS、TRACE 等请求方法。Body类请求包括但不限于POST、PUSH、PATCH、DELETE 等请求方法。

对于Url类请求，客户端是不能通过body写出数据的，服务端拿到的body是空的。

对于Body类请求，客户端通过body写进数据，服务端通过body接收数据。根据不同的enctype，数据可由@RequestBody或@RequestParam接收，但@RequestParam也是通过body接收。

接收请求

Java中绑定字段时键的格式

get方法接收数据

get方法编码类型为application/x-www-form-url

  • HttpServletRequest

    在方法的参数里加入HttpServletRequest request，然后request.getParameter(“name”)就可以获得value

  • 直接接收

    此时要求方法的参数名称要跟name名称一致；或者是用对象接收，传入对象对应的属性

  • @RequestParam(“name”)

    注解方法的参数，将值传到对应name标注的参数

post方法接收数据

  • HttpServletRequest

    同get方法

  • @RequestBody无参，注解方法的参数

    编码类型为application/json或application/xml时，必须使用**@RequestBody**处理。

    一次请求只能有一个Body，将body中的内容转为一个整体对象。请求体中的数据以json等格式传递（貌似没有简单的k-v格式）。
    要求接收数据的方法中 被@RequestBody修饰的参数 为String、Map、JSONObject或JavaBean数据类型。
    为Map时将内容绑定到单个K-V Map，内容拆分为其中的多个元素；为JavaBean时将内容绑定到单个对象中，需要保证内容的key和JavaBean的属性对应。

    由于get请求没有body所以只能用于post方式，所以@RequestBody不能用于get方式。

    注意，可以与url后面的参数一起用，即@RequestBody与@RequestParam同时使用，前者获取请求体的数据，后者获取url的数据

  • @RequestParam(“name”) @ModelAttribute也可以处理

    编码类型为application/x-www-form-urlencoded（form的默认类型）或multipart/form-data时，必须使用**@RequestParam(“name”)**处理，客户端将k-v对数据包装到body中。

    这是比较统一保守的规则，某些地方可能会支持部分特殊特性

    原理是通过servlet实现，参数放置在request.paramter，解析成map。

@RequestParam(“name”)实质是将Request.getParameter() 中的K-V Map利用Spring的转化机制ConversionService配置，转化成参数接收对象或字段。get方式中query String的值，和post方式中body data的值都会被Servlet接受到并转化到Request.getParameter()参数集中，所以@RequestParam都可以获取的到。

​ 有一个有趣的现象，如果Content-Type=mutipart/form-data，Body中加入参数和URL后面拼接参数一起做Post请求，都可以被加载到Param中，如果是同名的，只取用form-data中的。
​ 如果Content-Type=application/x-www-form-urlencoded和URL拼接的一起，如果是String类型，则两个值会被拼接，其他类型取的是URL拼接的参数。
