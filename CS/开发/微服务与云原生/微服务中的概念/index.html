微服务中的概念

Posted on 2024-01-02 | Post modified: 2024-01-04 | In 微服务与云原生

较全的细节参考

服务治理

对个服务集群中的服务进行管理

  • 负载均衡
  • 注册中心
  • 配置中心
  • 容错：访问服务时出现问题，通过容错机制切换到可用的服务
  • 限流，降级，熔断

网关

网关（服务网关，API网关）

网关 = 路由转发 + 过滤器

  • 路由转发：接收外界请求，转发到微服务上
  • 过滤器：权限校验、限流、监控、缓存、日志等

发展过程

微服务刚诞生的时候，只拆分了服务，客户端和各个服务进行交互

这样会有几个问题：

  • 客户端需要知道各个服务的ip地址，并且一个个请求
  • 每个服务都要重复实现认证、日志、缓存等公共逻辑
  • 若某些服务调用方式或参数发生改变，客户端需要做较大调整

所以引入网关，分割了内部和外部

  • 提供了外部请求的统一入口，统一对外暴露，内部由网关分发
  • 外部通过http，内部通过rpc

Zuul对比Spring Cloud Gateway

  • Zuul是netfix公司开源的第一代微服务网关；Gateway属于SpringCloud研发的第二代微服务网关
  • Zuul基于Servlet实现的，阻塞式的Api，不支持长连接；Gateway基于Spring5构建，能够实现响应式非阻塞式的Api，支持长连接
  • Gateway可以看做是一个Zuul 1.x的升级版和替代品，比其更高效，比Zuul 2更早使用Netty实现异步IO
  • Gateway明确区分了 Router 和 Filter，并且内置了非常多的开箱即用功能

Gateway对比Nginx

都可以实现对api的拦截，负载均衡、反向代理、请求过滤等

  • Gateway用Java写的，更适合结合微服务
  • nginx用C写的，性能更好

可以把Nginx当做第一层的网关

负载均衡

  1. 硬件负载均衡：采用专用的硬件设备来进行负载均衡，例如F5、Cisco ACE等。
  2. 软件负载均衡：采用软件程序来进行负载均衡，例如HAProxy、Nginx、Apache等。
  3. DNS负载均衡：通过DNS解析来进行负载均衡，例如使用DNS轮询或权重调度等方式。
  4. 防火墙负载均衡：利用防火墙设备进行负载均衡，例如使用Fortinet、Checkpoint等。
  5. 虚拟IP负载均衡：将多个物理设备组成一个虚拟设备，并通过共享一个IP地址来进行负载均衡，例如LVS、Keepalived等。
  6. CDN负载均衡：利用内容分发网络CDN来进行负载均衡，例如使用阿里云CDN、腾讯云CDN等。

Nginx

  • 反向代理：客户端只知道代理服务器的地址，不知道目标服务器的地址
  • 正向代理：服务器只知道代理服务器的地址，不知道客户端的地址

反向代理过程中，代理服务器会按照一定的规则，将客户端请求分发给不同的目标服务器进行处理，这一过程就是负载均衡

Nginx提供了六种负载均衡策略：

  • 轮询（默认）：请求会按时间平均分配给各服务器
  • 权重：在轮询的基础上指定权重
  • IP散列（ip_hash）：按照基于客户端IP的分配方式，这个方法确保了相同的客户端的请求一直发送到相同的服务器，以保证session会话。这样每个访客都固定访问一个后端服务器，可以解决session不能跨服务器的问题。
  • 最少连接数（least_conn）：把请求转发给连接数较少的后端服务器。轮询算法是把请求平均的转发给各个后端，使它们的负载大致相同；但是，有些请求占用的时间很长，会导致其所在的后端负载较高。这种情况下，least_conn这种方式就可以达到更好的负载均衡效果。
  • fair（第三方）：按照服务器端的响应时间来分配请求，响应时间短的优先分配
  • url_hash（第三方）：按访问url的hash结果来分配请求，使每个url定向到同一个后端服务器，要配合缓存命中来使用，可以使得同一个url（也就是同一个资源请求）会到达同一台服务器，一旦缓存住了资源，再此收到请求，就可以从缓存中读取。同一个资源多次请求，可能会到达不同的服务器上，导致不必要的多次下载，缓存命中率不高，以及一些资源时间的浪费。

服务注册与发现

  • 服务注册：**register(newServer)**的意思，将某个服务的模块信息（通常是ip和端口，此外还可能有协议、版本号、运行环境等）注册到一个公共的组件上去（如zk）
  • 服务发现：**getAllServer()**的意思，即获取服务列表，服务新增和服务删减都能被其他调用者自动发现

服务注册和服务发现还需管理服务器的存活状态，即健康检查

需求

  1. 集群： 得组成集群，这样单台出现故障，不至于服务宕机，否则记录注册数据的机器挂了，注册信息都丢失了
  2. 数据同步： 组成了集群，得要对注册的信息进行数据同步，在一台机器A上注册了，在其他机器上也能看到
  3. 强一致性：数据同步，在多台要有一致性的要求，保证数据不会出现不一致的情况
  4. 高并发高可用：要能保证请求量比较大的情况下，服务还能保持高可用
  5. 选举机制： 在有集群和数据同步以及一致性要求的情况下，得有一个master来主持整个运作，那就要有选取机制，确保选举公平和稳定
  6. 分布式：微服务上云，需要支撑分布式上的不同环境的机器互联

常用注册中心

  • Nacos：CP或AP，默认AP，使用Raft算法
  • ZK：CP，使用ZAB算法
  • Eureka：AP，
  • Consul：CP，使用Raft算法
  • ETCD：CP，使用Raft算法

对比参考1，参考2

对于注册中心，可用性A的要求高于一致性C，可以接受注册中心返回几分钟前的旧信息，但是不能接受没有返回信息

CAP理论

  • 一致性(Consistency)：所有节点在同一时间具有相同的数据
  • 可用性(Availability) ：保证每个请求不管成功或者失败都有响应
  • 分区容错(Partition tolerance) ：系统中任意信息的丢失或失败不会影响系统的继续运作

CA：传统单机数据库；AP：分布式系统，例如很多NoSQL

一致性协议算法：Paxos、Raft、ZAB

Raft是Paxos的简化版，ZAB是专门为ZK设计的协议，借鉴了Paxos算法

配置中心

配置中心的作用

  • 配置实时生效：兼顾实时性和一致性

    先前想要动态配置需要使用数据库，对数据库进行轮询

  • 配置管理：权限管控，多环境（开发和生产），灰度发布，版本管理和回滚，配置格式校验（json、yaml），监听查询（查询使用了配置的实例）等

参考

常用配置中心

  • Apollo
  • Nacos
  • Spring Cloud Config
  • Disconf（3年没维护了）
  • 非专业配置中心：Zookeeper和ETCD，缺乏权限、灰度发布、版本控制、审核等功能

熔断与降级

指标监控

监控项：

  • cpu、内存、磁盘、网络、io
  • 线上系统的jvm gc
  • 线上系统的各个接口的QPS和延迟进行监控
  • 线上微服务系统的各个服务之间进行调用的调用链路

通常用prometheus和skywalking两个监控系统配合完成，prometheus可以对线上系统的服务器、jvm以及接口各个层次进行监控，skywalking可以追踪微服务调用链路
