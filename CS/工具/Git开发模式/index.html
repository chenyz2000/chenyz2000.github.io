Git开发模式

Posted on 2021-07-04 | Post modified: 2021-07-04 | In 工具

分支介绍

  • 圆点1所在的线为主线（master），方框指向的节点为每一个发布版本的标签（tag）

  • 圆点2所在的线为主要分支（develop）

  • 圆点3所在的线为新功能开发分支（feature）

  • 圆点4所在的线为新功能发布分支（release）

  • 圆点5所在的线为发布版本bug紧急修复分支（hotfix）

分支详细描述

  • master分支：用来记录官方发布轨迹
  • develop分支：一个集成分支，用来记录开发新功能的轨迹
  • feature分支：临时分支。每一个新的功能都应该创建一个feature分支，从develop分支中派生出来；当功能完成后，要合并回develop分支，合并后其生命周期即结束。feature分支不会和master分支有交汇。
  • release分支：临时分支。为预发分支，当需要发布时，应该将满足发布条件的feature分支合并到develop分支中，然后开出一个release分支，开始准备一个发布版本。这个分支上不能再添加新的功能，只有bug修复和该版本为导向的任务。一旦到了发布日期，release分支就要合并回master分支发布，并且打出版本标签，另外还需要合并回develop分支。
  • hotfix分支：临时分支。为线上bug修复分支，用来快速修复生产环境中出现的紧急问题。hotfix只会和master交互。

管理策略

develop分支必须从master的稳定版内容合并过来。

为了避免开发功能时开发人员相互干扰，所以用feature分支边开发边在本地调试修改，一个版本会有多个feature分支，且不断提交新的，覆盖原有的，当分支代码通过了代码审查，项目组长即可将此段代码合并入develop分支。

当所有需要的特性都继承进入develop分支时，即可发布release分支，对其进行测试和bug修复，完成后即可提交到master分支。

master上线过程中，一旦发现紧急问题，则进入hotfix分支，只针对某bug的紧急修复。

git提交规范

<type>:<subject>

其中，<type>为提交类别。feat表示新功能，fix修复bug，docs仅修改文档，perf改进性能，style修改格式，refactor重构，test增加或修改测试，revert回滚，merge合并

<subject>为简短描述，可以使用中文

多次提交，不要放在一起提交
