Git与GitHub

Posted on 2020-07-13 | Post modified: 2021-11-24 | In 工具

版本控制

  • 管理版本，随时回退到任何时刻

  • 协同开发，自动合并代码

创建分支

创建新分支git branch <newBranch>

创建新分支并切换git checkout -b <newBranch> [<Node>]，相当于git branch <newBranch>; git checkout <newBranch>

分支合并

当前分支A指向结点a，待合并分支B指向结点b

将待合并分支B并入当前分支A，使用命令：git checkout A; git merge B

  1. 若a和b无继承关系，则新结点c继承a和b，当前分支A指向结点c
    如下图，main指向c3，执行git checkout main; git merge bugFix获得如下的结果
  2. 若当前分支结点a是待合并分支结点b的父结点，则A指向到较新的结点b
  3. 若当前分支结点a是待合并分支结点b的子节点，则A比B已经较新，不改变指向

继承关系的父子结点中，父结点表示旧的提交，子结点表示新的提交

HEAD一般作为分支的指针，即HEAD→main→c1

HEAD也可以作为分支，即结点的指针，即HEAD→c1

main、新建分支都是分支，即节点的指针

结点哈希值、分支名、两种作用下的HEAD 都可以作为命令中用到的参数

变基

当前分支A指向结点a，目标分支B指向结点b

将当前分支A的修改汇入目标分支B，使用命令：git checkout A; git rebase B

  1. 若二者无继承关系，则新结点c继承b，A指向c，原A分支隐藏
  2. 若二者有继承关系，则使用rebase和merge道理相似。A要向B汇入；若A是B的父节点，则A更新指向到B；若A是B的子节点，则A已比B新，不更新指向

git rebase -i可以进行交互式变基，可以选择结点、调整顺序、调整信息等，不像原来的rebase只能默认全部变基

所以可以实现子结点向父结点变基，不会显示A比B新

git rebase B A，A分支向B分支注入

git checkout改变HEAD的指向

相对引用git checkout {x}{y}，x为结点或分支或HEAD，y为^或~表示父结点，y为~3表示上溯3个结点，这样可以不用输入哈希值

移动HEAD到某个结点git checkout <Node>

移动分支到某个结点git branch -f <Branch> <Node>

git reset <Node>将当前分支状态改到指定的Node

reset后，原本的变更还在，但是处于未加入暂存区状态

git revert <Node>在当前分支新建一个子结点，该结点状态与指定的Node之前的状态相同，即将Node之后记录取消（不修改历史提交，可以与他人同步）

git cherry-pick <Node1> <Node2> <Noden>，意思是择优挑选，将一些提交复制到当前位置后，按照Node1、Node2…Noden的顺序，HEAD←Node1←Node2←Noden，依次为子节点

修改先前提交的代码

会产生至少三条分支：原分支、修改分支、修改后的分支

  1. 用cherry-pick或rebase -i将原分支待修改的提交a，调整为修改分支中的最新结点a’
  2. 在修改分支中对a’进行修改
  3. 用commit --amend对a’进行提交获得a’’
  4. 将a’'和原分支中a以外的结点按照原来的顺序放入修改后的分支

远程分支，git checkout origin/<Branch>时会进入HEAD分离状态，只有远程的分支更新时origin/main的指向才会更新，不能直接在远程分支上进行操做

git fetch获取远程分支的更新，只更新origin远程仓库中的分支，不改变本地分支

git pull等于git fetch; git merge origin/main，先获取远程分支更新origin仓库，再和本地分支合并

git pull --rebase等于git fetch; git rebase origin/main

git push将本地分支推送到远程仓库，并更新origin仓库的分支指向

push是针对当前分支的吧，merge和rebase也是针对当前分支的

push可以带参数推送指定的分支

文件四个状态

初始状态是untracked

add后成staged

commit后成unmodified

有提交记录的文件修改后成modified

基本快照

git status查看当前仓库的状态信息，红色是未暂存，绿色是已暂存

暂存的文件修改后，状态会变成红色

git diff能查看已写入暂存和又改动却未写入暂存的区别

git reset可以取消暂存

commit只提交已暂存的文件

提交历史

git log

–oneline参数显示简洁版本，–graph显示分支与合并

查看每个commit的哈希值、提交者、时间、信息

哈希值很长，但是只要前七位就能锁定

版本回退，git reset commitID，该版本之后的版本不保留在当前版本控制中

有参数–hard不保留所有变更，–soft保留变更且变更内容处于staged状态，–mixed保留变更且变更内容处于modified状态

不过之后的版本也能通过git reflog查到所有commit的七位commitID

分支管理

创建分支

git branch <name>或git checkout -b <name> [template]

差别是后者切换到新分支

template是模板，默认是当前分支

分支建立后，与原分支平行，不会相互影响

查看分支

git branch查看本地分支

-r参数查看远程分支

-a参数查看本地和远程分支

删除分支

git branch -d <name>

切换分支

git checkout <name>

合并分支

合并的是commit的变更，不是对两个内容进行合并

原有分支如图

git merge <name>

将另一分支的不同的commit合并到当前分支

保留被合并的分支的记录

git rebase

变基，枚举变更的commit，continue依次变基

不保留被合并的分支的记录

rebase会让历史更简洁

远程仓库

添加删除远程仓库

git remote add <name> <url>

git remote rm <name>

查看所有远程仓库

git remote

推送到远程仓库

git push <name> <branch>

获取远程仓库更新

git fetch从远程获知，之后需要手动merge远程分支到当前分支

或git pull，相当于fetch+merge

切换时遇到The following untracked working tree files would be overwritten by checkout

解决方法一：git rm –cached

解决方法二：git clean -d -fx

GitHub

Folk与Pull Request

与Clone不同，Folk将别人的仓库复制一份到自己的账号，并与原本的仓库有所联系

在自己Folk下来的仓库创建一个分支，在此分支上修改

之后可以将此分支PR给原仓库，等待对方的审核

若通过，对方将commit合并到原仓库；若不通过，则继续修改提交而不需要PR

merge和rebase

  1. 下游分支更新上游分支内容的时候使用 rebase
  2. 上游分支合并下游分支内容的时候使用 merge
  3. 更新当前分支的内容时一定要使用 --rebase 参数
