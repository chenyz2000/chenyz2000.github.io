Git开发模式

Posted on 2021-07-04 | Post modified: 2023-05-11 | In 工具

git提交规范

<type>:<subject>

其中，<type>为提交类别。feat表示新功能，fix修复bug，docs仅修改文档，perf改进性能，style修改格式，refactor重构，test增加或修改测试，revert重做，merge合并

<subject>为简短描述，可以使用中文

多次提交，不要放在一起提交

实习中的实际开发流程

  1. 从clone下来的仓库的master分支拉出一个feature分支，在feature分支上开发
  2. feature分支推送到gitlab，在测试环境部署该feature分支
  3. 通过测试后，在gitlab上merge request（或称PR），等待代码审查人员的审核
  4. 通过code review后，feature分支合并到远程仓库的master分支
  5. 若未通过，仍可继续commit、pull，不需要重新提MR
  6. merge后，部署master分支到所有环境，这一步骤应当采取从部分部署到推全的过程
  7. 本地仓库拉取最新的提交

共同开发

当自己在拉出的feature分支开发时，别人修改了代码，master分支更新了

  1. 在本地的master分支，git pull拉取最新更新
  2. 切换到自己的feature分支，rebase到最新的master提交，git checkout feature-xx; git rebase master

这样会把feature分支放到master新的commit之后，并且HEAD仍指向feature最新的commit

开源项目的实际开发流程

  1. 对开源项目进行Fork，得到自己的远程仓库
  2. clone下来，直接在master分支上修改
  3. 提交修改，push到自己的远程仓库
  4. 提交Pull Request，请求将自己的仓库与原仓库合并

Fork是在原仓库的基础上创建一个新的分支（仓库），和创建分支非常类似，只是创建的分支是一个代码仓

共同开发

当开源项目更新了，应当对原仓库和Fork后的仓库两个remote进行操作

  • 添加上游仓库：从原仓库获取最新的更新，即创建一个新的remote，命名为upstream
  • 拉取上游仓库变动并合并：在本地进行rebase合并
  • 更新自己的远程仓库：push到Fork后到仓库，例如叫做origin

注意，这里操作的都是master分支，将本地的提交rebase到upstream最新的提交后

Git Flow工作流

分支介绍

  • master分支——主线（节点1所在的线）：用来记录官方发布轨迹，方框指向的节点为每一个发布版本的标签（tag）
  • develop分支——主要开发分支（节点2所在的线）：一个集成分支，用来记录开发新功能的轨迹
  • feature分支——新功能开发分支（节点3所在的线）：一个临时分支。每一个新的功能都应该创建一个feature分支，从develop分支中派生出来；当功能完成后，要合并回develop分支，合并后其生命周期即结束。feature分支不会和master分支有交汇。
  • release分支——预发布分支（节点4所在的线）：一个临时分支。为预发分支，当需要发布时，应该将满足发布条件的feature分支合并到develop分支中，然后开出一个release分支，开始准备一个发布版本。这个分支应当给测试人员测试，并且不能再添加新的功能，只有bug修复和该版本为导向的任务。一旦到了发布日期，release分支就要合并回master分支发布，并且打出版本标签，另外还需要合并回develop分支。
  • hotfix分支——发布版本bug紧急修复分支（节点5所在的线）：一个临时分支。为线上bug修复分支，用来快速修复生产环境中出现的紧急问题。hotfix只会和master交互。

管理策略

  • develop分支必须从master的稳定版内容合并过来。
  • 为了避免开发功能时开发人员相互干扰，所以用feature分支边开发边在本地调试修改，一个版本会有多个feature分支，且不断提交新的，覆盖原有的，当分支代码通过了代码审查，项目组长即可将此段代码合并入develop分支。
  • 当所有需要的特性都继承进入develop分支时，即可发布release分支，对其进行测试和bug修复，完成后即可提交到master分支。
  • master上线过程中，一旦发现紧急问题，则进入hotfix分支，只针对某bug的紧急修复。
